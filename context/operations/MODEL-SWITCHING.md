# Model Switching Planning Document

## What We're Building and Why

The `/v1/chat/completions` endpoint already accepts a `model` field in
every request — the API is OpenAI-compatible so clients always send one.
Today the server ignores it and always uses `GEN_MODEL` from `settings.py`
(`llama3.1:8b`), logging a warning when the names don't match.

The goal is to honor the requested model name so clients can select
any model pulled in Ollama, without restarting the stack.

---

## What Is Explicitly Out of Scope

- Pulling models on demand — the model must already be pulled in Ollama
- Per-model rate limiting or quota tracking
- Embedding model switching — `EMBED_MODEL` stays fixed; switching
  embedding models would invalidate all existing vectors in Qdrant
- Reranker model switching
- Exposing model selection in the watcher

---

## Required Changes

### 1. `api/ollama_client.py`

Add a `model` parameter to `generate()` and `stream_generate()`,
defaulting to `GEN_MODEL` for backwards compatibility:

```python
def generate(prompt: str, model: str = GEN_MODEL, timeout: float = 120.0) -> str:
    r = post("/api/generate",
             json={"model": model, "prompt": prompt, ...})

def stream_generate(prompt: str, model: str = GEN_MODEL, timeout: float = 120.0) -> Iterator[str]:
    with post("/api/generate",
              json={"model": model, "prompt": prompt, ...}) as resp:
```

### 2. `api/query_rag.py`

Add `model` parameter to `ask()` and `ask_stream_sync()`, pass it to
the generate calls:

```python
def ask(question: str, model: str = GEN_MODEL) -> str:
    ...
    answer = ollama_client.generate(prompt, model=model)

def ask_stream_sync(question: str, model: str = GEN_MODEL) -> Iterator[str]:
    ...
    yield from ollama_client.stream_generate(prompt, model=model)
```

### 3. `web/api_server.py`

Pass `req.model` through instead of discarding it. Remove the warning
log — a mismatched model name is now valid, not a misconfiguration:

```python
# _run_rag_with_timeout — add model param
future = loop.run_in_executor(_RAG_EXECUTOR, ask, question, model)

# _rag_stream_response — add model param
for text in ask_stream_sync(question, model):
    ...

# chat() endpoint
question = _extract_question_from_messages(req.messages)
model = req.model  # honor the client's requested model

if req.stream:
    return StreamingResponse(_rag_stream_response(question, model), ...)

answer = await _run_rag_with_timeout(question, model=model)
```

The response object should echo back the actual model name used so
clients can confirm what ran:

```python
"model": model   # was: GEN_MODEL (always the same)
```

## Open Questions / Decision Points

### Validation approach

**Option A — No validation (recommended to start)**
Forward whatever model name the client sends directly to Ollama. If the
model isn't available, Ollama returns a clear error (`model not found`)
which the API propagates as a 500. Simple, no extra round-trips.

**Option B — Validate against Ollama `/api/tags`**
Before running retrieval, call `GET /api/tags` and check the requested
model is listed. Return a 400 with a useful message if not. Costs one
extra HTTP request per query but gives a better client error.

**Option C — Validate against a config allowlist**
Only allow the models explicitly listed in `settings.py`. Reject
anything else at the API layer before touching Ollama. Simple to
implement, but requires a stack restart to add a new model.

Recommendation: start with Option A. If clients frequently send bad
model names and the 500 error is confusing, add Option B later.

### `/v1/models` endpoint

Currently lists only `GEN_MODEL`. Should be updated to return all
models pulled in Ollama (via `GET /api/tags`) so clients can discover
available models dynamically. This is a separate but related change.

## Success Criteria

1. `POST /v1/chat/completions` with `"model": "qwen2.5:14b"` returns an
   answer generated by `qwen2.5:14b`, not `llama3.1:8b`
2. Streaming and non-streaming paths both honor the model field
3. The response object echoes back the model name that was used
4. No model field → defaults to `GEN_MODEL` (no breaking change)
5. Unknown model → Ollama error surfaces as a 500 with the original
   error message (Option A) or a 400 (Options B/C)
6. `ollama_client.generate()` called without a `model` argument continues
   to work as before (default preserved)
